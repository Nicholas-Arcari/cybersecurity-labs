# SET Infectious Media Generator: Autorun USB Payload

> - **Phase:** Social Engineering - Infectious Media Generator - Physical Access Vector
> - **Visibility:** Medium - USB insertion is logged in system logs (Event ID 6416 for PnP); payload execution can trigger AV/EDR; no network traffic in the delivery phase (physical delivery)
> - **Prerequisites:** SET installed on Kali Linux; formatted USB drive; physical access to the target area for drive drop; Metasploit for the reverse shell listener; target with autorun enabled (Windows 7/legacy) or social engineering to induce manual opening
> - **Output:** Finding SE-005 (severity Medium); autorun.inf file + executable payload generated by SET; structure ready for copying to USB drive

- **Operating Environment:** Kali Linux (Attacker 192.168.0.110), Windows 7 VM (Victim 192.168.0.115)
- **Target:** Workstation with autorun enabled (simulated legacy environment)
- **Framework:** Social-Engineer Toolkit (SET) + Metasploit Framework
- **Documented Technique:** Infectious Media Generator - Autorun payload for removable media

---

## Executive Summary

SET's Infectious Media Generator automates the creation of payloads for removable media (USB drives) with autorun configuration. The technique exploits the Windows AutoRun/AutoPlay mechanism to automatically execute a payload when the USB device is mounted. Although AutoRun is disabled by default starting from Windows 7 SP1 (Microsoft Security Advisory 967940), the technique remains relevant in three contexts: legacy environments (unpatched Windows XP/7, SCADA/ICS industrial systems), environments with GPO policies that re-enable AutoRun, and scenarios where social engineering convinces the user to manually open the file.

In the lab, the autorun structure (file `autorun.inf` + executable payload) was generated and tested on a Windows 7 VM with AutoRun enabled. Inserting the USB drive triggered automatic payload execution, establishing a reverse shell to the attacker machine.

## Finding SE-005: Autorun Payload on USB Drive

**Finding ID:** `SE-005` | **Severity:** `Medium` | **CVSS:** 5.3

An attacker with physical access to the target area can deposit a USB drive containing a payload with autorun.inf. On systems with AutoRun enabled, the payload is automatically executed when the device is mounted. On modern systems, the user must manually open the file, but social engineering (deceptive filename, attractive USB label) can induce opening.

**PoC Scenario:** The analyst generates an autorun structure with Meterpreter payload via SET, copies it to a USB drive, and inserts it into the target Windows 7 VM. Autorun executes the payload and establishes a reverse shell.

## Impact and Remediation (Blue Team)

Payload execution via USB drive provides the attacker access to the target system without any network interaction during the delivery phase. The physical vector completely bypasses perimeter controls (firewall, email gateway, proxy).

### Recommended Countermeasures

1. **Disable AutoRun via Group Policy:** `Computer Configuration -> Administrative Templates -> Windows Components -> AutoPlay Policies -> Turn off AutoPlay: All drives`. This is the most effective and zero-cost countermeasure.
2. **USB Device Control:** solutions like Microsoft Defender for Endpoint Device Control, Symantec DLP, or Carbon Black to block or limit access to removable USB devices.
3. **Endpoint Protection:** AV/EDR with real-time scanning of removable media on mount. Payloads generated by SET are typically detected by standard signatures.
4. **Security Awareness:** specific training on the risk of found USB devices. The "USB drop" test (depositing drives in common areas and measuring the open rate) is a standard exercise in awareness campaigns.
5. **Physical Security:** access controls to work areas; policies prohibiting the use of personal USB drives on corporate workstations.

## Lab Experience

The test environment required a specific configuration: the lab Windows 10 VM had AutoRun disabled by default, making automatic execution impossible. To test the technique in its original form, a Windows 7 VM with AutoRun enabled was necessary.

File generation with SET was fast and automatic: the framework invoked msfvenom to create the Meterpreter payload and generated the `autorun.inf` file with the `open=payload.exe` directive. The entire operation took less than a minute.

Transferring files to USB required practical attention: on Kali the USB drive must be manually mounted (`mount /dev/sdb1 /media/usb`) and after copying, `sync` must be executed before unmounting to ensure complete data writing. A common lab error is forgetting the sync, resulting in corrupted files on the drive.

Testing on the Windows 7 VM confirmed automatic execution: on drive mount, Windows briefly showed the AutoPlay window and then executed the payload. The reverse shell was established in approximately 5 seconds. Detection by Windows Defender (even on the Windows 7 VM) was immediate - the payload was quarantined before execution. To complete the test, Defender had to be disabled, confirming that the technique in its raw form is now detected even on legacy systems.

An important observation: in a modern real engagement, the attacker would not use an .exe payload with autorun. Current techniques use .lnk (shortcut) files pointing to PowerShell scripts, or .iso files containing .lnk + .dll, exploiting the fact that Windows automatically mounts ISO images on double-click. The autorun.inf is a legacy technique that this lab documents for historical completeness and understanding of the underlying principle.

## Theoretical Analysis: Physical Vector and Social Engineering

The removable media attack exploits two distinct vulnerabilities:

1. **Technical vulnerability (AutoRun):** the Windows AutoRun mechanism automatically executes the file specified in `autorun.inf` when the device is mounted. Microsoft mitigated this vulnerability with Security Advisory 967940 (2011), disabling AutoRun for USB devices. However, AutoRun remains active for CD/DVD on many systems, and some organizations re-enable AutoRun via GPO for operational reasons.

2. **Human vulnerability (curiosity):** academic studies (Tischer et al., 2016 - "Users Really Do Plug In USB Drives They Find") demonstrated that 48% of USB drives deposited in public areas are inserted into computers within a few hours. Device labeling ("Salary_Report_2026", "Confidential_HR") significantly increases the open rate.

The combination of these two vulnerabilities makes the attack effective even when only one is present: on systems with AutoRun, the payload executes without interaction; on systems without AutoRun, social engineering can convince the user to manually open the file.

In the MITRE ATT&CK context, technique T1091 (Replication Through Removable Media) is classified as an Initial Access vector. Historically, it has been used in nation-state operations: Stuxnet (2010) initially propagated via USB drives in Iranian nuclear facilities, demonstrating that the physical vector can reach even air-gapped systems.

The Medium severity reflects the physical access requirement and AutoRun being disabled on modern systems. In specific environments (industry, critical infrastructure with legacy systems), the effective severity would be higher.
