# Wazuh SIEM & Suricata IDS - Home Lab Setup

This repository documents the installation, configuration and troubleshooting process of a security-monitored environment (SIEM) using **Wazuh** on Kali Linux and **Suricata** as IDS, with agents deployed on Windows machines.

The setup is designed to be flexible, working even in mobile environments (e.g. connection via Wi-Fi Hotspot).

---

### Architecture

Wazuh operates according to a hierarchical and modular client-server model.

Understanding this architecture is fundamental for effective deployment.

---

### Prerequisites

Installation on Kali Linux, often performed in virtualized environments or on non-server-grade hardware, requires careful resource planning.

The most common error leading to service failure (particularly the Indexer) is insufficient RAM, which triggers the Linux kernel's OOM (Out Of Memory) Killer process.

----------------------------------------------------------------------------------------
| Component | Minimum Requirement | Recommended Requirement | Technical Notes |
----------------------------------------------------------------------------------------
| CPU | 2 vCPU | 4-8 vCPU | The Indexer requires parallelism for shard management. Fewer than 4 cores can cause startup timeouts |
| RAM | 4 GB | 8-16 GB | The Indexer allocates by default 50% of RAM (up to 32GB) to the JVM Heap. With less than 4GB total, the system becomes unstable |
| Storage | 20 GB | 100 GB+ (SSD) | I/O speed is critical for indexing. Rotational disks can cause bottlenecks |
| OS | Kali Linux (Rolling) | Debian 11/12 Stable | Kali is based on Debian Testing. This means frequent library updates (libc, openssl) that may require pinning Wazuh packages |

---

### Central Components (Server Side - Kali Linux)

The "Wazuh Server" is not a monolith, but an orchestration of three distinct services that must communicate synchronously:

- Wazuh Indexer: A distributed full-text search and analytics engine, based on OpenSearch. Its function is high-speed indexing and persistent storage of security alerts generated by the Manager. Being Java-based (JVM), it is the most resource-intensive component.
- Wazuh Manager: The operational core of the platform. It receives logs from agents, decodes them, normalizes them and compares them against a rule set to detect anomalies or threats. It also manages communication encryption and endpoint inventory.
- Wazuh Dashboard: The web user interface (based on OpenSearch Dashboards) that allows analysts to visualize data, execute threat hunting queries and manage cluster configuration.

### Central Node Installation on Kali Linux

The following procedure assumes the user has root privileges on the Kali machine. Unlike Ubuntu or CentOS, Kali Linux does not always automatically start services after installation due to default security policies.

---

### System and Repository Preparation
The first step is to install the packages needed to manage secure HTTPS repositories and import the Wazuh GPG key. Using gnupg is mandatory to verify package integrity and prevent supply-chain attacks.

```Bash
apt-get update
apt-get install gnupg apt-transport-https curl lsb-release
```

GPG key import and repository addition:

```Bash
curl -s https://packages.wazuh.com/key/GPG-KEY-WAZUH | gpg --no-default-keyring --keyring gnupg-ring:/usr/share/keyrings/wazuh.gpg --import && chmod 644 /usr/share/keyrings/wazuh.gpg
echo "deb [signed-by=/usr/share/keyrings/wazuh.gpg] https://packages.wazuh.com/4.x/apt/ stable main" | tee -a /etc/apt/sources.list.d/wazuh.list
apt-get update
```

Analysis: Using signed-by in the `.list` file is a modern security best practice on Debian that binds that specific repository to that key, preventing a compromised key from validating packages from other repositories.

---

### Wazuh Indexer Installation and Configuration

The Indexer must be installed first.

```Bash
apt-get install wazuh-indexer
```

Critical Cluster Configuration: Modify `/etc/wazuh-indexer/opensearch.yml`. In an "All-in-One" (single node) environment, network configuration is vital.

```YAML
network.host: "127.0.0.1"  # Bind to localhost for security if direct external access to the Indexer is not needed
node.name: "node-1"
cluster.initial_master_nodes: ["node-1"]
plugins.security.ssl.http.enabled: true
```

JVM Memory Management: On Kali machines with limited resources (e.g. 4-8GB RAM), it is imperative to limit the Java Virtual Machine Heap to avoid crashes. Modify `/etc/wazuh-indexer/jvm.options`:

`-Xms2g` `-Xmx2g` This sets the minimum and maximum heap to 2GB, leaving residual RAM for the operating system and the Wazuh Manager.

Service Startup:

```Bash
systemctl daemon-reload
systemctl enable wazuh-indexer
systemctl start wazuh-indexer
```

Note: This step may take several minutes. If it fails, consult the TROUBLESHOOTING section for systemd timeout management.

---

### Wazuh Manager Installation

```Bash
apt-get install wazuh-manager
systemctl enable wazuh-manager
systemctl start wazuh-manager
```

After startup, verify that the manager is correctly listening on critical ports:

- `1514/TCP`: Secure communication with agents (enrollment and data).
- `55000/TCP`: RESTful API for management and interfacing with the Dashboard.

---

### Wazuh Dashboard Installation

```Bash
apt-get install wazuh-dashboard
```

The configuration of the `/etc/wazuh-dashboard/opensearch_dashboards.yml` file is where most connection errors occur. It is essential that the OpenSearch host URL exactly matches what is configured in the Indexer (HTTPS protocol and port).

```YAML
server.host: "0.0.0.0" # Listen on all interfaces to allow browser access
opensearch.hosts: ["https://127.0.0.1:9200"] # Internal connection to the Indexer
opensearch.ssl.verificationMode: certificate
```

### Agent Deployment: Windows and Raspberry Pi

SIEM effectiveness depends on correct agent configuration. The fundamental rule for stability is that the Agent version must be less than or equal to the Manager version.

### Windows Installation (Workstation/Server)

The user encountered difficulties with PowerShell installation. Let's analyze the correct command and its implications.

Optimized PowerShell Command:

```PowerShell
Invoke-WebRequest -Uri https://packages.wazuh.com/4.x/windows/wazuh-agent-4.11.2-1.msi -OutFile $env:tmp\wazuh-agent.msi; msiexec.exe /i $env:tmp\wazuh-agent.msi /q WAZUH_MANAGER='<ip_dispositivo_windows>' WAZUH_AGENT_NAME='WindowsPC' WAZUH_REGISTRATION_SERVER='<ip_dispositivo_windows>'
```

Parameter Analysis:

- `WAZUH_MANAGER`: The IP address of the Kali machine.
- `WAZUH_REGISTRATION_SERVER`: Often coincides with the Manager, but specifying it explicitly resolves issues where the agent doesn't know where to send the initial key request.
- `WAZUH_AGENT_NAME`: A unique identifier. If omitted, the PC hostname is used.

Post-Installation and Startup: After installation, the service does not start automatically in some configurations.

```PowerShell
NET START WazuhSvc
```

If the command returns "The requested service has already been started" but the agent does not appear in the Manager, the problem almost certainly lies in network connectivity (Windows Firewall or iptables on Kali blocking port 1514).

### Raspberry Pi Installation (Linux ARM)
Raspberry Pi devices use ARM architectures (armhf for 32-bit systems, aarch64 for 64-bit systems). The apt package manager handles architecture automatically, but pay attention to versions.

Architecture Verification:

```Bash
uname -m
# Expected output: 'aarch64' (Pi 3/4/5 with 64bit OS) or 'armv7l' (older Pi or 32bit OS)
```

Installation and Version Pinning: To prevent an automatic update (`apt upgrade`) from bringing the agent to a version higher than the Manager's (causing disconnection), it is vital to pin the package.

```Bash
curl -s https://packages.wazuh.com/key/GPG-KEY-WAZUH | gpg --no-default-keyring --keyring gnupg-ring:/usr/share/keyrings/wazuh.gpg --import && chmod 644 /usr/share/keyrings/wazuh.gpg
echo "deb [signed-by=/usr/share/keyrings/wazuh.gpg] https://packages.wazuh.com/4.x/apt/ stable main" | tee -a /etc/apt/sources.list.d/wazuh.list
apt-get update

# Explicit installation (replace IP_KALI with the actual IP)
WAZUH_MANAGER="IP_KALI" apt-get install wazuh-agent

# Version pinning (Hold)
echo "wazuh-agent hold" | dpkg --set-selections
```

### The "Agent 000" Concept (Manager Monitoring)

A recurring question concerns installing an agent on the Manager machine itself.

Technically, you should not install the wazuh-agent package on the Wazuh Manager server.

The wazuh-manager service natively includes a "local agent" module with ID 000.

This agent monitors the underlying operating system (Kali Linux) exactly like an external agent, but with direct privileges on local log files.

Attempting to install wazuh-agent on the same machine would cause conflicts on TCP port 1514 and on configuration files in `/var/ossec/etc/`, corrupting the installation.

Agent 000 configuration is done by directly modifying `/var/ossec/etc/ossec.conf`

### TROUBLESHOOTING.md: In-Depth Analysis and Error Resolution

This section addresses the specific issues encountered during configuration, providing a theoretical explanation of the cause and the technical resolution procedure.

### Critical Dashboard Error: "Wazuh dashboard server is not ready yet"

Symptom: When trying to access the web interface, the user sees an infinite loading message or the "server not ready" error. In the logs (`journalctl -u wazuh-dashboard` or `/usr/share/wazuh-dashboard/data/wazuh/logs/wazuhapp.log`), errors such as `ConnectionError`, `ECONNREFUSED 127.0.0.1:9200` or `Internal Server Error 500` appear.

Cause Analysis: The Dashboard acts as a frontend for the Indexer. This error indicates that the communication chain is broken. The main causes are:

- Indexer service not active: The Dashboard tries to connect to port 9200, but nothing responds.
- Protocol Mismatch (HTTP vs HTTPS): The Indexer uses HTTPS by default, but the Dashboard might be configured to use HTTP.
- Invalid or corrupted SSL certificates: The Indexer rejects the connection because the Dashboard does not present a valid certificate or the Dashboard does not trust the Indexer's CA.

Resolution Protocol:

Phase 1: Indexer Status Verification - Run on the Kali machine:

```Bash
systemctl status wazuh-indexer
```

If the status is failed (often with "`start operation timed out`"), proceed to the Section "Service Startup Error: 'Start operation timed out' (Systemd)". If it is active, verify that it actually responds to API requests:

```Bash
curl -k -u admin:PASSWORD https://127.0.0.1:9200
```

If you receive a JSON with the cluster status (e.g. "status" : "green" or "yellow"), the Indexer works. The problem is in the Dashboard configuration.

Phase 2: Dashboard Configuration Correction - Modify `/etc/wazuh-dashboard/opensearch_dashboards.yml`. Check the `opensearch.hosts` section. Incorrect: opensearch.hosts: [`http://127.0.0.1:9200`] Correct: opensearch.hosts: [`https://127.0.0.1:9200`]

Ensure that the CA certificate paths are correct and that the files exist and are readable by the wazuh-dashboard user:

```YAML
opensearch.ssl.certificateAuthorities: ["/etc/wazuh-dashboard/certs/root-ca.pem"]
opensearch.ssl.verificationMode: certificate
```

After every change, restart the service: `systemctl restart wazuh-dashboard`.

### Authentication Error: "Resource 'admin' is reserved"

Symptom: The user tries to change the admin user password through the Dashboard GUI (or API) and receives the JSON error: `{"status":"FORBIDDEN","message":"Resource 'admin' is reserved."}`.

Theoretical Analysis: In OpenSearch (and therefore Wazuh Indexer), the admin user is an internal "super-user" managed by the security plugin. For system integrity reasons, this user is marked as reserved and cannot be modified through the standard RESTful APIs that the GUI uses. The modification must occur at the backend configuration level, regenerating password hashes and reloading the security configuration.

Solution (Wazuh Passwords Tool): Wazuh provides a dedicated bash script for this operation, which automates the complexity of hash regeneration and updating linked configuration files (such as `filebeat.yml` which uses these credentials to send logs).

Navigate to the security tools directory:

```Bash
cd /usr/share/wazuh-indexer/plugins/opensearch-security/tools/
```

Run the `wazuh-passwords-tool.sh` script. To change the admin user password:

```Bash
bash wazuh-passwords-tool.sh -u admin -p NuovaPasswordSicura!
```

Crucial Note for Distributed Environments: Although the script attempts to update configurations, in some cases it is necessary to manually update the password in the `/etc/filebeat/filebeat.yml` file (`output.elasticsearch` section) and `/etc/wazuh-dashboard/opensearch_dashboards.yml` (`opensearch.password` section) as well, otherwise these components will lose access to the Indexer.

### Service Startup Error: "Start operation timed out" (Systemd)

Symptom: The wazuh-indexer service fails to start. systemctl status reports: `wazuh-indexer.service: start operation timed out. Terminating..`

Cause Analysis: Java Virtual Machine (JVM) initialization and OpenSearch cluster bootstrap are intensive operations. On limited hardware (such as a Kali VM with virtual disks or a Raspberry Pi used as a node), the time needed to reach the "Ready" state can exceed systemd's default timeout (usually 90 seconds). When the timeout expires, systemd sends a SIGTERM signal, killing the process just as it was about to start correctly.

Definitive Solution (Systemd Override): Retrying is not sufficient. It is necessary to extend or disable the timeout in the service unit file.

Create a systemd configuration override for the Indexer:

```Bash
systemctl edit --full wazuh-indexer.service
```

Locate the directive and add or modify the TimeoutStartSec parameter. Setting it to 0 disables the timeout (infinite wait), or set a high value (e.g. 600 seconds).

```Bash
TimeoutStartSec=0
```

Save, reload the daemon configuration and restart:

```Bash
systemctl daemon-reload
systemctl restart wazuh-indexer
```

This approach is often effective in lab environments or VMs with shared resources.

### Version Conflict: Agent Downgrade on Linux

Symptom: Agents on Raspberry Pi or other Linux systems disconnect after a system upgrade. Logs show negotiation errors.

Analysis: Wazuh guarantees compatibility only if Manager Version is greater than or equal to Agent Version. If an apt upgrade updates the agent to 4.10 while the manager is at 4.9, communication breaks. Downgrade is not directly supported by upgrade commands; it requires removal and reinstallation of the specific package.

Downgrade Procedure:

Uninstall the current version:

```Bash
apt-get remove wazuh-agent
```

Install the specific target version (e.g. 4.7.5-1):

```Bash
apt-get install wazuh-agent=4.7.5-1
```

Verify that the configuration (`ossec.conf`) and the key (`client.keys`) have been preserved (Wazuh usually does not delete configuration files upon package removal, but making a preventive backup is wise).

### Windows Agent Connectivity Issues (Firewall and Ports)

Symptom: The PowerShell installation on Windows completes successfully, the WazuhSvc service is running, but the agent does not appear in the Dashboard.

Analysis: If the service is active, the problem is almost always at the network level. Wazuh uses two critical ports:

- `1514 TCP`: Main channel for data and keepalive.
- `1515 TCP`: Enrollment channel (initial registration).

Diagnostic Procedure on Windows (PowerShell): Use the Test-NetConnection cmdlet (or .NET TCP objects) to verify if Windows can reach the Manager on port 1514.

```PowerShell
Test-NetConnection -ComputerName <ip_dispositivo_windows> -Port 1514
```

If the result is TcpTestSucceeded : False, the Kali Linux firewall is blocking the inbound connection or the Windows Firewall is blocking the outbound.

Resolution on Kali Linux: Check iptables or ufw.

```Bash
ufw allow 1514/tcp
ufw allow 1515/tcp
iptables -A INPUT -p tcp --dport 1514 -j ACCEPT
iptables -A INPUT -p tcp --dport 1515 -j ACCEPT
```

Note: Kali Linux often has restrictive default policies. Ensure that rules are persistent across reboots.

### Using agent_control

On the Manager, the `/var/ossec/bin/agent_control` command is the definitive tool to verify the real status of agents, bypassing potential Dashboard delays.

- `agent_control -l`: Lists all registered agents and their status (Active/Disconnected).
- `agent_control -i <ID>`: Provides extended details about a specific agent.

### Conclusions and Operational Recommendations

Implementing Wazuh on a hybrid architecture with Kali Linux as the central node provides a powerful security monitoring platform, but requires rigorous configuration management. The major criticalities lie in resource management (RAM for the Indexer), version synchronization between Manager and Agents, and correct SSL/TLS trust chain configuration.

Final Recommendations:

Package Pinning: On "rolling" systems like Kali and on distributed agents like Raspberry Pi, always pin package versions (apt-mark hold) once a stable configuration is reached.

Resource Monitoring: Use Agent 000 to monitor the health of the Kali server itself, configuring alerts for memory and disk usage.

Keystore Backup: Before every update, back up the .keystore files and certificates in `/etc/wazuh-indexer/certs` and `/etc/wazuh-dashboard/certs`.

Automatic Timeout Management: Systematically implement the TimeoutStartSec=0 override for the Indexer in virtualized or resource-limited environments to prevent false startup failures.
